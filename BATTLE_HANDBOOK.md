## 0. 这是什么 & 怎么连
- **AgenTank**：agent-first 坦克编程对战游戏。人类建坦克壳，你(agent)用 tank key 读写坦克的 JS 策略脚本、模拟、发布、打排位。
- **新坦克第一步**：拿到**新坦克自己的 tank key**（格式 `agtk_...`），先 `GET /api/agent/tank` 读取：`skillType`(技能!)、`latestCode`(当前代码)、`rankTier/rankScore`、maps、trainingBots。
- **API base**：`https://agentank.ai/api/agent/...`，所有请求带头 `Authorization: Bearer <TANK_KEY>`。
- **Agent 文档**：http://agentank.ai/agent-guide （建议先完整读一遍）。
- **核心端点**：
  - `GET /api/agent/tank` 读坦克+代码+技能+段位。
  - `POST /api/agent/tank/code` 发布新代码。body: `{code, notes, submittedBy, branch:"main"}`，**submittedBy 必填**(作者模型名)。
  - `POST /api/agent/tank/simulate` 沙盒对战(不影响排名)。body: `{opponentId, mapId, code?}`。省略 code 用已发布代码。
  - `POST /api/agent/tank/challenge` **真实排位**(影响ELO/rank!)。body: `{randomOpponent:true, mapId:"random"}` 或 `{opponentTankId, mapId}`。返回含 `winnerTankId`、`challengerRankScore`、`rankChanges`。
  - `GET /api/matches/{matchUrlId}/agent.json?view=raw` 复盘(逐帧 records + map + meta)。`?view=events` 给关键事件摘要。
- **限流**：simulate 和 challenge **每 2 秒一次**(共享)。脚本里 sleep 2.7~3s，偶发 "simulation cooldown active" 需重试。
- **发布约定**：分析/模拟可自由做；**发布真实代码前最好经用户确认**(除非用户已授权持续迭代)。

## 1. 游戏机制（硬核实测，很多文档没明说，极重要）
1. **子弹速度 = 2 格/帧；坦克 = 1 格/帧**(boost时2格/帧)。→ 直射只有在 **贴脸(≤2格)** 或 **敌人沿你的射线轴移动(同行/同列且顺着打)** 时才稳中；对**中远距横穿**的敌人直射必空(它1帧离开车道，子弹还没到)。
2. **战斗 128 帧硬上限**。到点按**星星数量**判胜负。
3. **两种且仅两种获胜方式**：① 子弹击毁敌人；② 128帧到点星星多者胜。
4. **同归于尽由星数判胜**：双亡时星多者赢(领先→赢，落后→输，平→存疑可能平/看excitement)。→ **只有星领先时才值得换命对射**。
5. **草丛 `o` 隐藏坦克**：坦克站/移动在草丛上时，对敌**隐身**(敌方读到的 enemy.tank = null)。可用于伏击。
6. **`enemy.stars` 可读**(敌方星数！)、**`me.stars`**、**全局变量跨帧可保存**(`globalThis.__x`)、**`game.bombs` 可读**(避雷/预警)。
7. **施放技能消耗当帧动作**：`me.boost()/teleport()`+`turn()`+`fire()` 同帧不成立，只会施放技能然后挨打。赢对枪要么"上一帧已激活加速(免费转向)"，要么"已经正对敌人(预瞄)"。
8. **teleport**：CD40，**不转向**，落点距敌≤4格锁火2帧，不能落星上(重定向邻格)，落后2帧不能拾星。优先用于**逃生**(瞬移到距敌≥5的安全掩体格，根治贴脸被秒)。
9. **敌方子弹只在我方前方90°视野且无遮挡时可见** → 侧后方来弹看不见，"看到子弹再躲"太晚；把"敌对齐+有视线+正对我"当作即将开火来预判。
10. **地图** `map[x][y]`(坐标是 `[x,y]` 数组)：`x`墙 / `m`土堆(可被子弹摧毁, 挡路挡弹挡视线) / `o`草(隐身,可穿) / `.`空。
11. **段位/分**：定级后 胜≈+20 / 负≈-20；青铜0-300/白银300-600/黄金600-900/白金~900-1200/钻石~1200-1500/大师~1500-1800/王者(champion)顶端 #1≈2978。约45%对局由星数决定，55%由击毁。
12. **入口**：`function onIdle(me, enemy, game)`；每帧默认执行1个排队动作；**onIdle只在队列空时被调用——若某帧不排任何动作会空转→runTime判负！每帧必须至少排一个动作(如 turn)。**

## 2. 当前最强策略 = v17（分层决策，已与天梯冠军meta对齐）
`onIdle` 决策顺序（每层命中就 return）：
1. **拦截射击**：若"当前朝向开火"能命中敌人(逐帧模拟子弹2格/帧 vs 敌人沿朝向1格/帧的碰撞)，且不是"0星必输对射"→ 开火。
2. **战斗**：敌在我清晰射线上→ 正对则开火(星数感知:落后/平不换命)；未正对则转向瞄准；被贴脸且敌已瞄我→ 传送逃生/侧移。
3. **闪避**：可见来弹威胁→ 传送逃生/侧移出弹道。
4. **抢星**：BFS最短路径去星(无路则挖土`m`开路)；安全时可技能冲刺；避免走进敌人已瞄准的格子(safeStep绕行)。
5. **兜底**：巡逻。
- 技能自适应：`typeof me.teleport/me.boost === 'function'` 判断当前技能用哪套。
- **命中纪律(goodShot)**：只打 贴脸≤2 或 沿轴 的必中枪，不空放(空放会让唯一子弹在飞、无法再开火)。
- 完整代码见本机 `/home/finance/tank_game/v17.js`(若新对话在同机可直接复用；否则按上面结构重写)。辅助函数：canShoot(视线)、directionTo、turnDirection、delta/add/isOpen/manhattan、bfsFirst(带节点上限600的BFS)、interceptShot(碰撞预判)、teleportEscape、goodShot、stepOffLane/evade(掩体侧移)。

## 3. 天梯前10 制胜 meta（数据研究结论，务必内化）
分析 top-15 中 6 个坦克 17 场对局(含 myth 家族调试日志逆向真实决策树)：
- **顶尖不是纯蹲草丛！** 移动率 **~48%**(每隔一帧就动)，草丛占用仅 ~13%(有些0%)。纯静止蹲草的会**输**。
- 真实决策频率：`抢星(213) ≈ 占车道(203) > 技能连杀 > 争星(79) > 抢完星撤退(24)`；纯蹲草状态仅~1%(收尾用)。
- **制胜核心 = 车道锁定伏击(lane-lock)**：站在**一条直线车道上的草丛格**(隐身)，朝**空车道预开火**，敌人走进"看不见的飞行子弹"被杀。草丛只是让伏击者隐形的工具。
- **极致开火纪律**：只打同轴必中枪(命中率>30%)，抢完星**立即撤离暴露格**(post-star-retreat)。
- **星纪律**：只抢**我方更先到**的星；对手更近的争不到就放弃，转去占车道，别走进埋伏。
- **技能当连杀引擎**(如stun: setup→commit→followup)，并**识别敌方流派**(护盾/冰冻/双弹/加速)调整距离。
- 一句话：**抢星经济 + 车道同轴射击 + 抢完撤退 + 草丛只用于隐身伏击**；移动约半数，别纯蹲。

## 4. 已验证失败/无效的战术（别再走弯路！ELO实测）
| 战术 | 结果 | 教训 |
|---|---|---|
| 炸弹主动埋雷(throwBomb) | 45%回退 | 浪费动作帧 |
| 领先时进攻性传送贴近敌人 | 44%回退 | teleport不转向+锁火，进攻性差 |
| 草丛全车道避让(敌隐身时避开草线) | 40%大回退 | 草丛太普遍→避让面积过大，严重拖累抢星 |
| 星数感知对射(领先才换命) | ~中性 | 逻辑对但不移动ELO(可保留) |
| 超时前保星领先(晚期避战) | ~中性 | 想法对但不涨分 |
| **重度蹲草丛(v26-v29)** | 回退 + **runTime** | 蹲守→拖满128帧→每帧onIdle太重→累计超时判负；且数据证明纯蹲会输 |
> **元教训**：任何**降低抢星效率**或**浪费动作帧**的添加都回退。edge = 星效率+存活+命中。改进方向 = 让核心**更快更准**，不是加复杂规避。

## 5. 代码编写经验 / 踩坑（runtime 是头号杀手）
- **每帧必须排一个动作**(go/turn/fire/skill)；不排→onIdle空转→**runTime判负**。想"原地不动"就 `me.turn(...)`(转向不移动，仍隐身在草上)。
- **runTime 预算**：onIdle 别太重。**每帧都跑 BFS + 拦截模拟**在128帧长局会**累计超时**。对策：① 静态数据(草丛连通分量)用 `globalThis` **缓存只算一次**；② 用**扁平数组**别用字符串键对象做大循环；③ BFS 加节点上限(如600)并命中即返回；④ 只在"真的要移动/交战"时跑重逻辑，蹲守/等待帧要极轻。
- **坐标是 `[x,y]` 数组**，不是 `{x,y}`。`me.tank.position[0]` 是 x。写错会让移动/瞄准/BFS全崩。
- **技能施放消耗当帧动作**：想靠技能赢对枪必须提前一帧激活或已预瞄。
- **enemy.tank/enemy.bullet/game.star 可能为 null**(敌隐身/无星)，每处判空。
- **teleport 落点**必须合法(在图内、非墙/土/敌格/敌弹格)，否则失败仍耗CD；逃生落点选距敌≥5(免锁火)且无敌方射线(掩体)。
- 发布前先 `simulate` 打训练bot(nova-scout/azure-hunter/crimson-bastion) 跑 classic/arena + 几张社区图，确认无 runTime/error、无明显回退。

## 6. 测试方法论（纪律 = 法则①数据驱动）
- **每个改动**：部署后 `challenge` grind **≥40-50场**看 rank/RS 变化。**回退就立刻移除**，不恋战、**不连续叠加未验证的改动**(会污染归因)。
- **一次只测一个变量**。
- 本机工具(可复用/照抄)：
  - `sim.py`：沙盒模拟。`python3 sim.py batch <codefile|published> --maps=classic,arena`，从 replay meta.result.winner 判胜(0=我方胜)。
  - `challenge.py N`：打N场真实排位，输出每场结果+MATCHES(便于复盘)+RS变化。sleep 2.7~3s防限流。
  - `review.py <matchUrlId...>`：复盘(双方 shots/hit/star/skill + 关键事件)。
  - **探针技巧**：`me.speak(str)` 会进回放(action="say")，可用来打印 `enemy.stars`、`game.bombs`、`globalThis` 值来验证机制。
- **信息与策略池** `POOL.md`：人机共维护，每条带置信度★。每批复盘后更新"哪些信息对/错、哪些策略工作/不工作"。
